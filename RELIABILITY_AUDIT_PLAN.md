# FicTrac Reliability Audit And Implementation Plan

Date: 2026-05-01

Scope:

- native FicTrac fork in this tree
- reliability, installability, and day-1/day-2 lab usability
- minimum-surface-area fixes first, broad rewrites only where justified

This document turns the current audit into an implementation backlog. Each item includes:

- validated problem statement
- evidence in the current source tree
- why the issue matters operationally
- optimal minimum-surface-area solution
- concrete implementation fix
- suggested acceptance check

The goal is not to redesign FicTrac from scratch. The goal is to make this fork dependable for lab users while changing as little as possible in the native codebase per step.

## Design Principles

1. Prefer additive guardrails over architectural rewrites.
2. Fix failure reporting before adding new behavior.
3. Make setup failures happen early, explicitly, and with actionable messages.
4. Preserve existing tracking behavior unless a change directly improves correctness or recoverability.
5. Ship a supported install path per platform rather than a collection of tribal-knowledge steps.

## Observation 1: Recorder Shutdown Can Drop Final Messages And Outputs

### Validated Problem

The asynchronous recorder can exit before it has drained its queued writes.

Evidence:

- `src/Recorder.cpp` sets `_active = false` in the destructor before the queue is drained.
- `src/Recorder.cpp` runs the worker loop as `while (_active)`, so a shutdown request stops the loop even if messages remain queued.
- `src/Trackball.cpp` uses `Recorder` for the `.dat` file, video-frame sidecar log, socket output, serial output, and some raw-frame artifacts.
- `src/Logger.cpp` also uses `Recorder` for the log file, so the most useful terminal diagnostics can be lost near shutdown.

Current code paths involved:

- `src/Recorder.cpp`
- `src/Trackball.cpp`
- `src/Logger.cpp`

### Why This Matters

This is a direct reliability defect, not a style issue.

Failure modes:

- final data rows missing from the `.dat` file
- final log lines missing when diagnosing a run failure
- socket or serial consumers seeing truncated end-of-run messages
- raw-frame sidecar logs not matching saved artifacts after a stop or exception

For a hardware-facing experiment tool, partial output with no explicit failure marker is one of the worst outcomes because it creates silent ambiguity.

### Minimum-Surface-Area Solution

Keep the asynchronous recorder design, but change shutdown semantics from:

- `stop immediately when inactive`

to:

- `stop accepting new messages, then drain all queued messages, then exit`

This is a local fix in `Recorder` and does not require redesigning `Trackball`, `Logger`, or the output formats.

### Implementation Fix

1. Add an explicit recorder state split:
   - `accepting_messages`
   - `worker_running`

2. Change `addMsg` to reject new messages only after shutdown begins.

3. Change the worker loop to continue while either of these is true:
   - recorder is still accepting messages
   - queue is non-empty

4. In the destructor:
   - stop accepting new messages
   - notify the worker
   - join the worker only after it drains the queue

5. If any low-level write fails, capture a sticky error flag and print one final synchronous `stderr` message during teardown.

Files to touch:

- `include/Recorder.h`
- `src/Recorder.cpp`

### Suggested Acceptance Check

- unit-style test or small integration test that enqueues many messages and destroys the recorder immediately, then asserts all lines landed in the file
- manual stop test while running FicTrac and verify the final `.dat` row count and final log lines are present

## Observation 2: Config Parsing Is Permissive, Lossy, And Can Rewrite Files Destructively

### Validated Problem

The config parser accepts malformed input too quietly and rewrites config files too aggressively.

Evidence:

- `src/ConfigParser.cpp` silently skips lines without a colon.
- duplicate keys overwrite prior values with no warning.
- the file itself contains `FIXME: replace corresponding keys rather than overwriting the whole file!`
- `write()` rewrites the config from the internal map rather than preserving original ordering and formatting.
- `write()` uses a fixed `char tmps[4096]` buffer with `sprintf`.
- `Trackball` writes parameters back to the config automatically during construction.

Current code paths involved:

- `include/ConfigParser.h`
- `src/ConfigParser.cpp`
- `src/Trackball.cpp`
- `src/ConfigGUI.cpp`

### Why This Matters

This creates operational brittleness in the exact area users touch most:

- hand-edited configs
- config files regenerated by the GUI
- debugging why a setup behaves differently after being reopened

Failure modes:

- malformed lines are ignored rather than producing a hard validation error
- duplicate keys produce hidden last-one-wins behavior
- comments and ordering can shift unexpectedly after a write
- user intent is harder to reconstruct when something breaks

### Minimum-Surface-Area Solution

Do not replace the config format yet.

Instead, keep the current key/value syntax and make the parser stricter and the writer safer.

The minimum useful upgrade is:

- strict read validation with line numbers
- duplicate-key warnings or hard failures behind a mode
- non-destructive write behavior for unchanged keys where possible
- explicit `validate` mode for configs without launching tracking

### Implementation Fix

Phase 1:

1. Track line numbers during parse.
2. Emit parse diagnostics with file name and line number.
3. Detect duplicate keys and report them clearly.
4. Replace `sprintf` with a bounds-safe formatting path.
5. Stop auto-writing the config on every tracking startup unless the process is explicitly in a config-mutating mode.

Phase 2:

6. Add a parser result object containing:
   - warnings
   - errors
   - duplicate keys
   - normalized resolved values

7. Add a CLI validation path that reads the config and exits after reporting issues.

Files to touch:

- `include/ConfigParser.h`
- `src/ConfigParser.cpp`
- `src/Trackball.cpp`
- `exec/fictrac.cpp`

### Suggested Acceptance Check

- malformed config line fails validation with line number
- duplicate key emits a deterministic warning or error
- running normal FicTrac does not rewrite the config unless a flag requests it
- existing sample config still validates and runs unchanged

## Observation 3: Build Defaults Hide Warnings And Favor Risky Optimization Settings

### Validated Problem

The current build config suppresses warnings and applies aggressive optimization defaults.

Evidence:

- `CMakeLists.txt` uses `/w` and `/WX-` on MSVC release builds.
- `CMakeLists.txt` uses `-w`, `-Ofast`, and `-march=native` on non-Windows builds.
- there is no supported warning-enabled CI configuration.

### Why This Matters

For legacy C++ that interacts with threads, cameras, codecs, sockets, and binary buffers, warnings are one of the cheapest reliability tools available.

Failure modes:

- undefined behavior is harder to catch early
- floating-point semantics can change under `-Ofast`
- binaries become less portable under `-march=native`
- contributors cannot tell whether a change introduces new warnings because the build hides them

### Minimum-Surface-Area Solution

Do not chase full warning cleanliness across the entire codebase first.

Instead:

- keep current release behavior if needed for parity
- add a safer default for supported packaging builds
- add an opt-in strict or developer build preset

This avoids destabilizing the tracking core while still making the codebase maintainable.

### Implementation Fix

1. Introduce CMake options or presets for:
   - `fictac_warnings` or `FICTRAC_ENABLE_WARNINGS`
   - `FICTRAC_PORTABLE_BUILD`
   - `FICTRAC_STRICT_BUILD`

2. For portable builds:
   - remove `-march=native`
   - replace `-Ofast` with `-O2` or `-O3`

3. For developer CI:
   - enable warnings without `-Werror` initially
   - surface them in CI logs

4. Keep a separate path for any parity-sensitive release if required.

Files to touch:

- `CMakeLists.txt`
- optional new `CMakePresets.json`

### Suggested Acceptance Check

- portable preset builds on at least one Windows machine and one Linux machine without local tuning
- warning-enabled CI job reports warnings visibly
- parity test remains unchanged for the validated Windows binary if release flags are intentionally preserved there

## Observation 4: Install Scripts Are Manual, Fragile, And Not A Lab-Safe Support Story

### Validated Problem

The current install path depends on user judgment and interactive shell handoffs.

Evidence:

- `install_windows.ps1` prompts for the MSYS path interactively
- `install_windows.ps1` asks the user to copy a package install command into an external MSYS shell
- `install_windows.ps1` mutates the user `PATH`
- both install scripts delete and recreate the build directory
- `install_ubuntu.sh` assumes Ubuntu 20.04 or 22.04 and performs an in-place source build without packaging

### Why This Matters

This is exactly the kind of flow that works for the maintainer and fails for everyone else.

Failure modes:

- users install mismatched dependency versions
- local environment changes are hard to undo
- reinstall behavior varies between machines
- lab onboarding becomes dependent on one person remembering the right sequence

### Minimum-Surface-Area Solution

Stop treating install as a shell recipe and start treating it as a supported product surface.

The minimum credible cross-platform story is:

- one documented supported path on Windows
- one documented supported path on Linux
- reproducible dependency resolution
- a self-test after install

### Implementation Fix

Recommended direction:

1. Replace ad hoc install scripts with CMake presets plus platform-specific bootstrap wrappers.
2. Keep wrappers non-interactive by default.
3. Make wrappers accept explicit dependency roots through flags or environment variables.
4. Add a post-build self-test that runs the bundled sample parity or at least a sample smoke run.
5. Produce a packaged artifact:
   - Windows: zipped release bundle or installer containing the executable, runtime DLLs, sample config, and a self-test command
   - Linux: tarball package or documented package-manager route plus the same self-test

Possible implementation shapes, ordered from smallest to broadest:

- smallest: improve current scripts and add presets
- medium: ship release bundles from CI
- broader but cleanest: package through CPack or a lab-owned distribution repo

Files to touch:

- `install_windows.ps1`
- `install_ubuntu.sh`
- `CMakeLists.txt`
- optional `CMakePresets.json`
- CI workflow files

### Suggested Acceptance Check

- fresh Windows machine can build or install without interactive shell copying
- fresh Ubuntu machine can build or install from documented commands only
- both paths end with a self-test command that returns pass/fail clearly

## Observation 5: There Is No Native Preflight Or Doctor Command

### Validated Problem

The current CLI only exposes run-time options for verbosity, stats, and source override.

Evidence:

- `exec/fictrac.cpp` advertises `CONFIG_FN`, `--verbosity`, `--stats`, and `--src`
- there is no native `validate`, `doctor`, `preflight`, or `self-test` command
- many fatal checks occur only after substantial setup has already started in `Trackball`

### Why This Matters

The gap between “config exists” and “experiment is safe to start” is where most lab-user pain lives.

Users need a fast answer to:

- is this config syntactically valid?
- can the source be opened?
- is camera mode compatible with the selected source settings?
- are output paths writable?
- are socket and serial endpoints available?

Without preflight, all of those problems surface late and inconsistently.

### Minimum-Surface-Area Solution

Add a read-mostly preflight path to the native CLI rather than building a new app.

Suggested command shape:

- `fictrac --validate config.txt`
- `fictrac --preflight config.txt`
- optional `fictrac --self-test`

`validate` should be side-effect-free.

`preflight` may probe devices and output paths, but should not start tracking.

### Implementation Fix

1. Extend CLI parsing in `exec/fictrac.cpp` with:
   - `--validate`
   - `--preflight`
   - `--self-test`

2. Split current `Trackball` startup into staged checks:
   - parse config
   - resolve source
   - open source and query dimensions/fps
   - validate output targets
   - validate optional socket/serial configuration
   - only then start grabber/tracker threads

3. Extract a lightweight startup report object with pass/fail plus warnings.

4. Print a short human-readable report and optionally a machine-readable JSON summary.

Files to touch:

- `exec/fictrac.cpp`
- `include/Trackball.h`
- `src/Trackball.cpp`
- `include/ConfigParser.h`
- `src/ConfigParser.cpp`

### Suggested Acceptance Check

- bad config fails under `--validate` without opening a camera
- missing output directory fails under `--preflight` before tracking starts
- sample config passes `--validate` and `--preflight`

## Observation 6: Calibration UI Is A Setup Bottleneck And Already Documents Its Own Flakiness

### Validated Problem

The configuration GUI still carries explicit hacks around rendering and user interaction.

Evidence:

- multiple `cv::waitKey(100)` calls are annotated with `FIXME`
- comments explicitly describe a `dirty hack` to make images draw reliably
- there is still TODO/FIXME debt around how clicks are captured

Primary file involved:

- `src/ConfigGUI.cpp`

### Why This Matters

Calibration is the first hands-on contact most users have with FicTrac.

If calibration feels laggy, nondeterministic, or easy to mis-click, users correctly infer that the system is brittle.

### Minimum-Surface-Area Solution

Do not redesign the GUI stack first.

The minimum win is to stabilize the existing interaction model and make it easier to recover from user mistakes.

### Implementation Fix

1. Remove forced timing hacks where possible by making redraw and input sequencing explicit.
2. Add a visible status footer or prompt state so the user always knows what click is expected next.
3. Add undo/reset for the current calibration step.
4. Add a non-interactive summary screen before write-back showing exactly what will be saved.
5. If the GUI still proves flaky, add a headless config assistant script outside the core runtime rather than expanding GUI complexity in C++ first.

Files to touch:

- `src/ConfigGUI.cpp`
- optional helper docs in `README.md` and `doc/params.md`

### Suggested Acceptance Check

- users can recalibrate the sample config without needing repeated key mashing
- accidental mis-clicks can be undone without restarting the whole flow

## Observation 7: Automated Validation Is Too Narrow For A Hardware-Facing Legacy Tool

### Validated Problem

Automated coverage currently centers on one useful but narrow offline sample parity test.

Evidence:

- only one native test file exists: `tests/test_sample_parity.py`
- current tests validate the canonicalized sample `.dat` signature and timestamp-column handling
- CI builds the code but does not run the parity test or broader smoke tests

### Why This Matters

The current test does protect a valuable invariant: tracking output parity on the bundled sample video.

But the most brittle code paths are elsewhere:

- config validation
- recorder teardown
- output creation failures
- camera-mode startup logic
- shutdown after stream end

These are exactly the bugs that regress quietly.

### Minimum-Surface-Area Solution

Keep the parity test, then add tests specifically around the failure boundaries that have the highest user pain and the lowest mocking burden.

### Implementation Fix

Priority order:

1. Add recorder drain tests.
2. Add config-parse validation tests.
3. Add startup/preflight tests on the sample config.
4. Add smoke tests for output-path failures and duplicate keys.
5. Run the parity test in CI on Windows.

Files to touch:

- `tests/test_sample_parity.py`
- new tests under `tests/`
- CI workflow files

### Suggested Acceptance Check

- CI fails on recorder drain regressions, malformed configs, and sample parity regressions
- local contributors can run the same checks with one documented command

## Observation 8: Runtime Diagnostics Need A Structured Run Summary

### Validated Problem

Current diagnostics are log-centric and dispersed across startup and runtime messages.

Evidence:

- `Logger` writes lines asynchronously to a timestamped log file
- startup checks are distributed inside `Trackball` construction
- there is no structured manifest summarizing which subsystems initialized successfully

### Why This Matters

When a user says “FicTrac failed,” the first support question should not be “please send me a screenshot and your entire terminal history.”

There should be one small artifact that answers:

- what binary version ran
- which config file was used
- whether source open succeeded
- whether data log, socket, serial, raw, and debug outputs opened
- why the run ended

### Minimum-Surface-Area Solution

Add one structured run summary JSON alongside current logs. Do not replace the text log.

### Implementation Fix

1. Create a small run manifest object populated during startup and shutdown.
2. Record:
   - version and build date
   - source type and resolved source identifier
   - effective fps and fps mode
   - output targets requested and opened
   - termination reason
   - frame counts if available
3. Write the manifest on clean exit and on early startup failure where possible.

Files to touch:

- `exec/fictrac.cpp`
- `include/Trackball.h`
- `src/Trackball.cpp`
- optional logger or misc helpers

### Suggested Acceptance Check

- every run produces a concise machine-readable summary or an explicit startup-failure summary

## Observation 9: Cross-Platform Install Should Become A Product Surface, Not A Maintainer Ritual

### Validated Problem

There is no clean, modern, cross-platform install story today.

Evidence:

- Windows relies on MSYS and interactive copy/paste
- Linux relies on an Ubuntu-specific source-build script
- there are no CMake presets
- there is no packaged release artifact or post-install self-test contract

### Why This Matters

This is separate from build correctness. A project can build on the maintainer's machine and still be operationally un-installable for everyone else.

### Minimum-Surface-Area Solution

Introduce a release contract before introducing a packaging ecosystem.

Recommended contract:

- one build preset per supported platform
- one release bundle per supported platform
- one self-test command included in each bundle

This keeps the scope bounded and still produces a much better user experience.

### Implementation Fix

Phase 1:

1. Add `CMakePresets.json` for Windows and Linux.
2. Add CI jobs that build with those presets.
3. Archive release bundles from CI.

Phase 2:

4. Add a packaging step:
   - CPack, or
   - simple release bundling scripts if CPack is too invasive

5. Bundle:
   - executable(s)
   - required runtime DLLs or shared-library expectations
   - sample config
   - sample data if licensing permits
   - self-test command docs

Phase 3:

6. Evaluate package-manager distribution only after the release bundle path is stable.

Potential outputs:

- Windows zip or installer
- Linux tarball or distro-specific package

Files to touch:

- `CMakeLists.txt`
- new `CMakePresets.json`
- CI workflow files
- optional bundling scripts under `scripts/` or repo root

### Suggested Acceptance Check

- a new user can install on Windows or Linux from a release artifact and verify with one self-test command

## Recommended Execution Order

This is the order that gives the best reliability gain per unit of code churn.

1. Recorder drain fix.
2. Config parser hardening plus no-implicit-rewrite behavior.
3. Native `--validate` and `--preflight` command.
4. Structured run manifest.
5. CI expansion to run parity plus new startup tests.
6. Build presets and portable build cleanup.
7. Cross-platform release bundling.
8. Calibration GUI stabilization.

## Bootstrap And Preflight Proposal

This deserves explicit treatment because it unlocks day-1 usability.

### Bootstrap Goal

Answer: “Can this machine run FicTrac correctly?”

### Proposed Commands

- `fictrac --self-test`
- `fictrac --validate <config>`
- `fictrac --preflight <config>`

### `--self-test`

Should verify only repo-local or installation-local invariants:

- binary launches
- sample assets exist if bundled
- writable temp directory available
- sample parity or sample smoke run passes

### `--preflight`

Should verify per-run external dependencies:

- config parses cleanly
- source opens
- fps mode and trigger expectations are coherent
- output targets can be created
- serial and socket targets can be initialized if requested

### `--validate`

Should stay strictly side-effect-free:

- parse config
- report schema and value errors
- report duplicate keys and invalid combinations

## Cross-Platform Install Proposal

The install story should be divided into two supported paths.

### Path A: Users Install A Release Bundle

Best for most lab users.

Should include:

- binaries
- runtime dependencies
- sample config and self-test
- short install and verify instructions

### Path B: Developers Build From Source

Best for maintainers and CI.

Should include:

- CMake presets
- explicit dependency instructions
- one build command
- one test command

### Preferred Near-Term Implementation

1. Create CMake presets first.
2. Make CI build from those presets.
3. Emit release bundles from CI.
4. Keep current shell scripts only as compatibility wrappers around the new preset flow.

## Out Of Scope For The First Round

These may become worthwhile later, but they are not minimum-surface fixes:

- replacing the config format entirely with YAML or JSON
- rewriting the UI in another toolkit
- replacing the async recorder with a wholly different logging subsystem
- redesigning the tracking algorithm itself
